## 优点总结

1. **快速响应需求**：从 PDF → 网页 → PWA，4天内完成 7 次迭代，每次都基于用户真实反馈
2. **数据安全意识好**：全量使用 localStorage 本地存储，用户数据不上传任何服务器
3. **渐进式增强**：PDF → 静态网页 → 交互网页 → PWA，每一步都在前一步基础上叠加，没有推翻重来
4. **用户体验驱动**：勾选升星、连续打卡统计、热力图等都是基于"激励用户坚持"这个核心理念
5. **离线优先架构**：Cache First 策略确保断网也能正常使用
6. **零服务器成本**：纯前端 + GitHub Pages，完全免费部署

## 教训与改进点

1. **时间模板花了太多轮迭代才确定**：V1-V4 的时间安排都不合理（无缓冲、学习碎片化），应该在 V1 就跟轩哥确认好所有时间约束，而不是做出来再改。→ 下次开发前先做需求确认清单
2. **早期没考虑部署场景**：V3-V6 都在本地开发，到 V7 才考虑 PWA 和 HTTPS 部署。→ 新产品立项时就应规划部署方案
3. **文件冗余**：过程中产生了 schedule.html、schedule_app.html、index.html 三个版本共存的情况，容易混淆。→ 应在迭代过程中及时清理旧版本或统一命名
4. **功能膨胀风险**：V5 一次性加了太多功能（矩阵+拖拽+计时器+暗色模式+周总结+模板+导入导出+打卡），虽然都完成了但测试覆盖不够。→ 应控制单次迭代的功能范围，小步快跑
5. **git 习惯不好**：到部署时才初始化 git，前面 6 个版本的迭代过程完全没有版本管理。→ 新项目第一天就应 git init + commit
